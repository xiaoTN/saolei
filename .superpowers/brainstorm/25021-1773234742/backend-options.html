联机后端方案对比

先做局域网/本机，后续扩展公网。哪种架构最适合？

A

BroadcastChannel（同设备多标签页）+ 后续换 WebSocket

Phase 1：用浏览器原生 BroadcastChannel API，同一浏览器不同标签页之间实时同步状态，零依赖、零服务器。Phase 2：抽象出一个 Transport 接口，把 BroadcastChannel 换成 WebSocket，上层代码不变。

优点

  • Phase 1 完全无服务器，纯静态
  • 接口抽象后切换成本低
  • 开发调试极简单

缺点

  • Phase 1 只支持同设备，局域网两台机器不行
B

本地 Node.js WebSocket 服务器（局域网可用）

写一个极简 Node.js 脚本（约 80 行），跑在本地做信令和状态转发。局域网内任意设备扫码/输房间号可加入。后续部署到公网服务器只需改连接地址。

优点

  • 局域网内真实双设备可用
  • 和最终公网方案架构完全一致
  • 房间码逻辑在服务端，更可靠

缺点

  • 需要先 node server.js 启动服务
  • 比纯静态稍复杂一点
C

SharedArrayBuffer / SharedWorker（最底层共享内存）

通过浏览器 SharedWorker 或 SharedArrayBuffer 共享游戏状态，同设备标签页零延迟同步，不需要任何消息传递。

优点

  • 理论上延迟最低

缺点

  • 需要 COOP/COEP HTTP 头，纯静态部署麻烦
  • 和公网 WebSocket 方案差异很大，迁移成本高
  • 不推荐
